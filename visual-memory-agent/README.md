# Visual Memory Agent

Persistent visual memory for LibreChat using an MCP server + a Python API service + SQLite/file storage.

This agent is designed to:
- analyze images with a vision model,
- save structured visual memory,
- retrieve and search past visual references,
- version and compare image updates,
- and delete stored references when requested.

## Architecture

The implementation has two runtime components:

1. **MCP server (`mcp_server.py`)**  
   Exposes tools to LibreChat over MCP transport (SSE or stdio).  
   It does orchestration and forwards calls to the API service.

2. **Visual Memory API (`main.py`)**  
   FastAPI service that performs analysis, persistence, search, versioning, and deletion.

Supporting modules:
- **`database.py`**: SQLite schema, insert/query/search/version/delete helpers.
- **`storage.py`**: image decode/fetch/hash/save/load/delete helpers.

Data persistence:
- SQLite DB: `visual_memory.db` (table: `references_store`)
- File storage: `image_storage/`

---

## What Happens When a User Sends an Image

Default flow (important):

1. LibreChat calls MCP tool `analyze_image` (or `analyze_image_url`).
2. MCP calls API `/analyze-image`:
   - image is normalized into a data URI,
   - OpenRouter vision model is called with structured JSON output constraints,
   - result is parsed as rich JSON with:
     - `name`, `caption`, `summary_short`, `summary_detailed`
     - `ocr_text`
     - `layout`, `layout_regions`, `scene_type`, `style`
     - `objects`, `objects_detailed`
     - `topics`, `entities`, `key_phrases`
3. Because `VISUAL_MEMORY_AUTO_SAVE_REFERENCE` defaults to `true`, MCP then calls `/create-reference`.
4. API saves:
   - raw image file in `image_storage/`,
   - analysis fields in SQLite,
   - and returns `reference_id` + `version` + `reference_key` (e.g. `ref_001_v1`).

So yes: image analysis is model-generated, then persisted as structured memory.

---

## How the Agent Answers Later Questions About an Image

This is the key behavior:

- **`get_reference` / `fuzzy_search_references` do not re-analyze the image.**  
  They read stored fields from SQLite (`name`, summaries, OCR, layout, style, objects, topics/entities) and return them.

- The final assistant message is still generated by the chat LLM, but based on tool output context.

Why answers can feel short:
- Stored analysis may itself be concise.
- If follow-up asks are answered from stored JSON only, there is no fresh deep vision pass unless `analyze_image` is called again with a richer prompt.

## MCP Tools (What LibreChat Can Call)

## 1) `analyze_image`
Analyzes image input (`image_base64` or `image_url`) and returns structured JSON.  
If `VISUAL_MEMORY_AUTO_SAVE_REFERENCE=true`, it will auto-save even if `save_reference=false` is passed by mistake.

Inputs:
- `image_base64?`
- `image_url?`
- `prompt?`
- `model?`
- `save_reference?` (default from env, usually `true`)
- `reference_name?`
- `user_id?`

Returns:
- Rich analysis JSON (`name`, `caption`, `summary_short`, `summary_detailed`, `ocr_text`, `layout`,
  `layout_regions`, `scene_type`, `style`, `objects`, `objects_detailed`, `topics`, `entities`, `key_phrases`)
- If saved, includes `reference` metadata (`reference_id`, `version`, `reference_key`, image URL/markdown)

## 2) `analyze_image_url`
Same as `analyze_image`, but URL-first convenience wrapper.

## 3) `create_reference`
Stores an analyzed image as a new reference (`v1`) with duplicate detection by image hash.

Returns:
- `reference_id`, `version`, `reference_key`, `is_duplicate`, message, image URI/URL

## 4) `create_new_version`
Adds a new version under an existing `reference_id` (e.g., `v2`, `v3`, ...).  
Also performs duplicate hash checks.
If duplicate exists under another reference and `move_if_duplicate=true` (default), it moves that stored record into the target reference as the next version.

## 5) `get_reference`
Fetches one stored reference by `reference_id` and optional `version`.  
If `version` is omitted, latest version is returned.

## 6) `fuzzy_search_references`
Searches references by free text over:
- `name`
- `caption`
- `summary_short`
- `summary_detailed`
- `ocr_text`
- `layout`
- `style`
- `objects`
- `objects_detailed`
- `topics`
- `entities`
- `key_phrases`

Uses score-based matching only (no broad fallback injection of unrelated recent records).

## 7) `compare_versions`
Loads two saved image versions and asks the vision model for structured diffs (`added`, `removed`, `changed`, etc.).

## 8) `delete_reference`
Deletes selected references (or all references) from DB and removes associated files.
Supports:
- `reference_ids` to delete full references (all versions)
- `reference_keys` to delete specific versions (e.g. `ref_001_v2`)
- `delete_all=true` to wipe all

## Stored Schema (`references_store`)

Main fields:
- `reference_id` (logical ID, e.g. `ref_001`)
- `version` (`v1`, `v2`, ...)
- `reference_key` (derived at runtime in API responses, e.g. `ref_001_v2`)
- `name`
- `image_hash` (SHA-256, duplicate detection)
- `image_uri` (path to stored file)
- `caption`
- `summary_short`
- `summary_detailed`
- `ocr_text`
- `layout`
- `layout_regions`
- `scene_type`
- `style`
- `objects` (JSON-serialized array in DB, parsed back to list on read)
- `objects_detailed`
- `topics`
- `entities`
- `key_phrases`
- `analysis_json` (full normalized payload for richer downstream responses)
- `previous_version_id`
- `user_id`
- `created_at`

Constraints:
- unique `(reference_id, version)`
- unique `image_hash`

## Environment Variables

From `.env.example`:

- `OPENROUTER_API_KEY`
- `OPENROUTER_MODEL` (default: `qwen/qwen2.5-vl-72b-instruct`)
- `OPENROUTER_SITE_URL`
- `OPENROUTER_APP_NAME`
- `OPENROUTER_TIMEOUT_SECONDS`
- `VISUAL_MEMORY_API_BASE_URL` (MCP -> API URL, default `http://127.0.0.1:8090`)
- `VISUAL_MEMORY_API_TIMEOUT_SECONDS`
- `VISUAL_MEMORY_PUBLIC_BASE_URL` (for image URLs returned to clients)
- `MCP_TRANSPORT` (`sse`, `streamable-http`, or `stdio`)
- `MCP_HOST`
- `MCP_PORT`
- `MCP_ACCESS_TOKEN` (required for `sse`/`streamable-http`; checked from `Authorization: Bearer <token>`)

Also used in code:
- `VISUAL_MEMORY_AUTO_SAVE_REFERENCE` (default `true`)
- `AGENT_DB_PATH` (SQLite DB location, default `./visual_memory.db`)
- `IMAGE_STORAGE_DIR` (file directory, default `./image_storage`)

## Interaction Patterns

### New image memory
`analyze_image` -> auto `create_reference` -> assistant replies with summary + reference metadata.

### Ask about previously saved image
`get_reference` (or `fuzzy_search_references` then `get_reference`) -> assistant answers from stored fields.

### Update an existing image
`analyze_image` (new image) -> `create_new_version`.

### Compare old vs new
`compare_versions` with `reference_id`, `from_version`, `to_version`.

### Remove memory
`delete_reference` with `reference_keys` (specific versions), or `reference_ids` (whole references), or `delete_all=true`.

---

## Why Responses May Be Brief (and How to Improve)

If follow-up answers are too short, common reason is that the stored analysis is short.  
The assistant is often summarizing stored memory, not running a fresh deep vision pass.

To get richer responses:
- Use a more detailed `prompt` during `analyze_image` (domain-specific extraction instructions).
- Re-run `analyze_image` for deeper follow-up questions.
- Include explicit user asks like: “Give full OCR, UI hierarchy, and detailed object list.”

## Notes

- Duplicate images are detected by SHA-256 hash.
- Image files are served statically from `/image_storage`.
- Returned image URLs include `?h=<image_hash>` cache-busting tokens to prevent stale image rendering.
- `name` is auto-filled when missing (fallback to summary/caption/object-derived title).
- `objects`, `topics`, `entities`, `key_phrases`, and `layout_regions` are normalized to lists when returned.
- If OpenRouter returns HTTP `429`, `/analyze-image` now returns a full structured fallback payload instead of a hard failure so saving/searching can continue.
- When all rows are deleted from `references_store`, the SQLite autoincrement sequence is reset so new rows start from `id=1`.
